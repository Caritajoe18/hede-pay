# Developer Feedback: Hedera Agent Kit Integration

## 1. Token Management and "Plugin Bloat"

### The Issue
The `allCorePlugins` configuration, while convenient, attaches the descriptions of every available tool to the system prompt for every request. 
- **Cloud-Based Models:** For developers using free-tier models, this massive token overhead frequently triggers `rate_limit_exceeded` errors because system prompts consume the majority of the Tokens Per Minute (TPM) quota.
- **Local Models:** For developers running models locally on resource-constrained hardware (e.g., an Intel Mac with 8 GB of RAM), this high context window overhead severely degrades inference speeds and causes system lag.

### Feedback
The documentation should strongly emphasise manual plugin selection for development environments. Highlighting this optimization helps developers save context window space, prevent rate limits, and drastically reduce latency on local models.

---

## 2. TypeScript Framework Type Mismatches

### The Issue
Developers frequently encounter `No overload matches this call` compiler errors when wiring up tools. This is caused by type mismatches between the tool arrays generated by the core Hedera Agent Kit and the strict type expectations of destination orchestration frameworks (such as LangChain's `StructuredTool`). If the developer omits manual type casting (`as StructuredTool[]`), the project fails to compile.

### Feedback
The SDK documentation should provide clear, framework-specific code examples showcasing how to correctly initialize and cast tools. Highlighting the explicit separation between helper libraries (e.g., `@hashgraph/hedera-agent-kit-langchain`) and providing clear mapping templates will eliminate these common integration roadblocks.

---

## 3. Parameter Hallucination in Smaller Models

### The Issue
Smaller local models (such as `llama3.2:3b`) frequently struggle with parameter formatting and placeholder injection. For example, when invoking the `get_account_query_tool` to check a user's own balance, smaller models often pass raw placeholder strings (like `"yourAccountId"`) or format the JSON payload incorrectly, resulting in `400 Bad Request` or `402` errors from the Mirror Node.

### Feedback
The toolkit would benefit greatly from an automated **Schema Guard** or a pre-configured prompt injection template. If a query is self-referential (e.g., "What is my balance?"), the SDK should automatically fallback or defaults to the active `operatorId` from the initialized Hedera Client, shielding smaller models from logic formatting errors.
